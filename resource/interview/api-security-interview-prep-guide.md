# API Design & Security Interview Prep Guide

## 1. REST API Design Best Practices

### Resource Naming
- Use plural nouns for collections (/orders)
- Nested resources: /orders/{id}/items
- Never use verbs in URL paths
- Keep URLs simple and intuitive

### HTTP Methods
- GET: Retrieve resource data safely
- POST: Create new resource instance
- PUT: Full update of existing resource
- PATCH: Partial update of resource
- DELETE: Remove resource permanently

### Status Codes
- 200 OK: Successful request completed
- 201 Created: Resource created successfully
- 204 No Content: Success with no response body
- 400 Bad Request: Invalid client input
- 401 Unauthorized: Authentication required
- 403 Forbidden: Authenticated but not allowed
- 404 Not Found: Resource does not exist
- 409 Conflict: Request conflicts with state
- 422 Unprocessable: Valid syntax, semantic error
- 429 Too Many Requests: Rate limit exceeded
- 500 Internal Server Error: Server failure

### Pagination Strategies
- Page-based: Use page number and size params
- Cursor-based: Prefer for large datasets
- Include total count in response metadata
- Set reasonable default page sizes

### Filtering Approach
- Use query parameters for filters
- Example: ?status=active&date_from=2024-01-01
- Avoid complex query languages in URLs
- Support sorting with sort parameter

### API Versioning
- URL path versioning: /v1/orders (simplest)
- Header-based: Accept header with version
- URL path is more visible and cacheable
- Header-based keeps URLs cleaner

### HATEOAS Principle
- Include related resource links in responses
- Reduces client-side URL hardcoding
- Helps clients discover available actions
- Not always necessary for simple APIs

## 2. Idempotency

### Definition
- Same request produces same result always
- Regardless of how many times executed
- Critical for retry-safe operations
- Prevents duplicate processing issues

### Critical Use Cases
- Payment processing endpoints
- Order creation operations
- Any operation where retry is possible
- Network timeout scenarios common

### Implementation Strategy
- Client sends idempotency key in header
- Key is typically a UUID generated by client
- Server checks if key was already processed
- Return cached response if key exists

### Storage Mechanism
- Use Redis for idempotency key storage
- Set TTL to 24 hours typically
- Key format: idempotency_key value
- Value stores response or status

### Processing Flow
1. Check if idempotency key exists in Redis
2. If exists, return cached response immediately
3. If not, process the request normally
4. Store result with key before returning
5. Return response to client

### Practical Implementation
- Every payment endpoint requires idempotency
- Order creation must be idempotent too
- Client sends X-Idempotency-Key header
- Check Redis first before processing
- Prevents double charges on network retry
- TTL of 24 hours balances safety and cleanup

## 3. API Security

### OWASP API Security Top 10

#### API1: Broken Object Level Authorization
- Access other user's data by changing ID
- Validate ownership on every request
- Implement proper authorization checks
- Don't trust client-provided IDs blindly

#### API2: Broken Authentication
- Weak token implementation issues
- No rate limiting on login endpoints
- Missing multi-factor authentication
- Session management vulnerabilities

#### API3: Broken Object Property Authorization
- Expose sensitive fields in responses
- Return admin fields to regular users
- Filter response based on user role
- Use DTOs to control exposed fields

#### API4: Unrestricted Resource Consumption
- No rate limiting on expensive endpoints
- Missing pagination limits
- Allow unlimited file uploads
- Implement request size limits

#### API5: Broken Function Level Authorization
- Regular user calls admin endpoints
- Missing role-based access checks
- Verify permissions for each function
- Implement RBAC properly

#### API6: Unrestricted Business Flows
- No rate limit on signup process
- Abuse of promotional codes
- Implement business logic validation
- Monitor for unusual patterns

#### API7: Server-Side Request Forgery (SSRF)
- Attacker controls server-side requests
- Validate and whitelist target URLs
- Don't fetch arbitrary user-provided URLs
- Use allowlists for internal services

#### API8: Security Misconfiguration
- Default configurations left unchanged
- Verbose error messages in production
- Missing security headers
- Regular security audits needed

#### API9: Improper Inventory Management
- Undocumented endpoints exposed
- Test APIs accessible in production
- Old API versions still active
- Maintain API inventory regularly

#### API10: Unsafe API Consumption
- Trust third-party data without validation
- Missing input sanitization
- Validate all external data sources
- Implement proper error handling

### Input Validation
- Validate all inputs on server side
- Never trust client-side validation alone
- Whitelist allowed values when possible
- Reject unexpected or malformed input

### Rate Limiting
- Implement per-user rate limits
- Add per-endpoint rate limiting
- Consider per-IP rate limiting too
- Return 429 with Retry-After header

### CORS Configuration
- Restrict allowed origins specifically
- Limit allowed HTTP methods
- Specify allowed headers explicitly
- Never use Allow-Origin: * in production

## 4. OAuth2 & JWT Deep Dive

### OAuth2 Flows

#### Authorization Code Flow
- User authenticates with auth server
- Server redirects with authorization code
- Backend exchanges code for access token
- Best for web applications with backend

#### Authorization Code + PKCE
- Adds code challenge and verifier
- Prevents authorization code interception
- Required for mobile and public clients
- Enhanced security for SPA applications

#### Client Credentials Flow
- Service exchanges credentials for token
- Uses client_id and client_secret
- For service-to-service communication
- No user context involved

#### Resource Owner Password Flow
- Deprecated and insecure
- Never use in production systems
- Migrate to Authorization Code flow
- Direct password handling is risky

### JWT Structure
- Header: Algorithm and token type
- Payload: Claims with user data
- Signature: Validates token integrity
- Base64url encoded three parts

### Token Lifecycle Management

#### Access Token
- Short-lived typically 15 minutes
- Contains user permissions and claims
- Sent in Authorization header
- Used for API resource access

#### Refresh Token
- Long-lived typically 7 days
- Used to obtain new access tokens
- Store securely in httpOnly cookie
- Rotate on each refresh operation

#### Token Rotation
- Issue new refresh token on each use
- Invalidate old refresh token immediately
- Prevents token reuse attacks
- Detect compromised tokens quickly

### Signing Algorithms
- HS256: Symmetric signing, simpler setup
- RS256: Asymmetric signing, more secure
- RS256 allows verification without secret
- Use RS256 for production systems

### Practical Implementation
- Use RS256 for all production APIs
- Auth service signs with private key
- All services verify with public key
- Access tokens expire in 15 minutes
- Refresh tokens rotate on each use
- Service-to-service uses client credentials
- Short-lived tokens for services (5 min)

## 5. API Gateway Patterns

### Gateway Responsibilities
- Route requests to backend services
- Authenticate and authorize requests
- Implement rate limiting policies
- Transform request and response formats
- Cache responses when appropriate
- Log all incoming requests

### Implementation Options
- Spring Cloud Gateway for Java apps
- Kong for high-performance needs
- Nginx for simple reverse proxy
- AWS API Gateway for cloud-native

### Gateway vs BFF Pattern
- Gateway: Shared across all client types
- BFF: Backend for Frontend per client
- Web BFF optimizes for web clients
- Mobile BFF aggregates for mobile needs

### When to Use Gateway
- Always use in microservices architecture
- Centralizes cross-cutting concerns
- Simplifies client-side complexity
- Provides single entry point

### Practical Implementation
- Spring Cloud Gateway as entry point
- Handle JWT validation at gateway
- Rate limiting per tenant or user
- Request logging for audit trails
- Route to downstream services
- Mobile clients use separate BFF
- BFF aggregates multiple API calls

## 6. Common Interview Questions

### Q1: How do you design REST API?
- Use plural nouns for resources
- Apply correct HTTP methods
- Implement pagination properly
- Version your API from start
- Use consistent error format

### Q2: What is idempotency importance?
- Same result regardless of retries
- Prevents duplicate charges or orders
- Implement with idempotency keys
- Store keys in Redis with TTL

### Q3: Explain OWASP API Top 10
- Broken object level authorization
- Broken authentication mechanisms
- Unrestricted resource consumption
- SSRF and security misconfigurations
- Implement proper validation always

### Q4: How do you implement OAuth2?
- Authorization Code + PKCE for web
- Client Credentials for services
- Never use Resource Owner Password
- Implement proper token rotation

### Q5: How do you handle token rotation?
- New refresh token on each use
- Invalidate old refresh token
- Prevents replay attacks effectively
- Monitor for suspicious activity

### Q6: Gateway vs direct service calls?
- Gateway for external client access
- Centralized auth and rate limiting
- Direct calls for internal services
- Reduces latency for internal calls

### Q7: How do you handle API versioning?
- URL path for simplicity and visibility
- Header-based for cleaner URLs
- Support multiple versions temporarily
- Deprecate old versions gradually

### Q8: What is CORS importance?
- Cross-origin resource sharing
- Restricts which origins can access
- Whitelist specific trusted origins
- Never allow all origins in production

### Q9: How do you design error responses?
- Consistent structure across API
- Include error code and message
- Add details field for debugging
- Use RFC 7807 ProblemDetail format

### Q10: How do you prevent brute force?
- Rate limit login attempts strictly
- Lock account after failed attempts
- Add CAPTCHA for suspicious activity
- Monitor and alert on patterns
